## 3. API Routes

### `POST /api/chat`

**Input:**
```ts
{
  personaId: string;
  message: string;
  history?: { role: "user" | "assistant"; content: string }[]; // last 6 sent
}
```

**Output:** `text/event-stream`. Events, in order:
```
data: {"type":"emotion","emotion":"calm"}\n\n
data: {"content":"word "}\n\n         ← repeated, one per token/fragment
data: {"type":"error","message":"..."}\n\n   ← only on failure, then [DONE]
data: [DONE]\n\n
```
**Critical:** the client (`chat/[id]/page.tsx`, `call/[id]/page.tsx`) only reads `parsed.content` for text — never emit `{type:"token", token}`, it renders nothing.

**External calls:** Groq `chat.completions.create({ stream: true })` via `groq-sdk`. Falls back to a 4-line hardcoded stub (`stubStreamResponse()`) if `useStub` is true.

```ts
const useStub = !process.env.GROQ_API_KEY || process.env.RUNPOD_OFFLINE === "true";
```

Builds a 5-zone system prompt server-side every turn: Zone 0 (who user is, from `users.displayName`/`profileBio`), Zone 1 (persona identity from `characterCardText`), Zone 2 (hardcoded "talk like a real person" naturalness rules + `[emotion]` tag instruction), Zone 3 (Pinecone `queryMemories` top-3), identity reminder. History: last 6 turns appended to `messages[]`, not baked into system text.

SSE parsing note: the Groq SDK already parses chunks into JS objects — no raw-byte `\n\n` buffering needed server-side (unlike the old RunPod fetch). The `[emotion]` tag is regex-extracted from the start of the stream (`/^\s*\[(\w+)\]\s*/`) and stripped before the first content chunk is emitted; `stripStrayLeadingChar()` guards against the model fusing the tag with the next character.

**Env required:** `GROQ_API_KEY`, `GROQ_MODEL` (optional, defaults `llama-3.1-8b-instant`), `RUNPOD_OFFLINE` (stub toggle), `PINECONE_API_KEY` (optional — degrades to `[]` memories if absent).

### `POST /api/tts`

**Input:**
```ts
{ personaId: string; text: string; emotion?: string } // emotion defaults "default"
```

**Output:**
```ts
// success
{ audio_base64: string; sample_rate: number }
// failure
{ error: string } // with 400/401/404/422/502 status
```
422 specifically = persona has no `voiceRefB64` set.

**External calls:** `POST https://api.runpod.ai/v2/{RUNPOD_TTS_ENDPOINT_ID}/runsync`
```ts
body: { input: { text, voice_b64, exaggeration, cfg_weight, temperature } }
// RunPod wraps response: { output: { audio_base64, sample_rate } }
```
Chatterbox params resolved: persona `voiceParamsJson` overrides > `CHATTERBOX_PRESETS[emotion]` > `.default`. `voiceRefB64` has its `data:audio/...;base64,` prefix stripped before sending (Python `base64.b64decode` chokes on it).

`AbortController` timeout: **600_000ms (10 min)** — covers a RunPod cold start (~5-8 min). Comment explicitly flags Phase 5+ should switch to `/run` + poll `/status/{id}` instead of blocking one request that long.

**Env required:** `RUNPOD_API_KEY`, `RUNPOD_TTS_ENDPOINT_ID`. `RUNPOD_OFFLINE` does **not** affect this route — TTS always calls RunPod regardless.

**Cost control (comment block at top of file):**
```
// RunPod TTS endpoint: lzgcc945pqi103
// Set max_workers=0 on dashboard when not testing to stop billing.
// Set max_workers=1 before any TTS test session.
// Cold start: ~5-8 min. Warm generation: ~3-30s per clause.
```

### `GET /api/deepgram-token`

**Input:** none (auth via session cookie).

**Output:**
```ts
{ token: string; expiresIn: number } // token ~40 chars alphanumeric
// or
{ error: string }
```

**External calls:** `POST https://api.deepgram.com/v1/projects/{DEEPGRAM_PROJECT_ID}/keys`
```ts
body: { comment: "temp-browser-key", scopes: ["usage:write"], time_to_live_in_seconds: 60 }
// response: { key: string, api_key_id: string }
```

**Why not `/v1/auth/grant` (JWT):** browsers can't set a custom `Authorization` header on a WebSocket handshake. The workaround is passing the token via the `Sec-WebSocket-Protocol` subprotocol array (`new WebSocket(url, ["token", key])`), but that header has a length limit — grant-token JWTs (~485 chars) get silently rejected; a short project API key (~40 chars) fits. This was a real bug found and fixed mid-project (see §11 Critical Lessons).

**Rate limit:** RunPod... no — **Deepgram** limits project-key creation to **250/day**. The call page mints one per mount; React Strict Mode is disabled specifically to avoid doubling this on every dev hot-reload.

**Env required:** `DEEPGRAM_API_KEY` (Member+ role), `DEEPGRAM_PROJECT_ID`.

### `POST /api/memory/commit`

**Input:** `{ personaId: string; transcript: string }` (transcript = `"role: content\n"` lines joined).

**Output:** `{ stored: number; facts?: string[] }` or `{ stored: 0, stub: true }` or `{ stored: 0, error: "extraction failed" }`.

**External calls:** RunPod LLM completions (**not Groq** — see §9), asks it to extract up to 5 facts as a JSON array. Inserts into `memoriesLog` (Postgres) with `source: "call"`. Comment: `// Phase 7: also upsert to Pinecone and store pineconeId` — **not implemented**, facts land in Postgres only, never in the `persona-memories` Pinecone index that `/api/chat`'s `queryMemories()` reads from.

**Env required:** `RUNPOD_API_KEY`, `RUNPOD_LLM_ENDPOINT_ID` — **both commented out in `.env` post-Groq-migration**, so this route always takes the stub branch now. See §9.

### `POST /api/personas/generate-card`

**Input:** `{ personaId: string }`.
**Output:** `{ characterCardText: string; stub: boolean }`.
**External calls:** RunPod LLM completions (**not Groq**), same env-var problem as memory/commit — always falls back to `buildStubCard()` (a hardcoded generic card) now. Errors are caught and logged, never surfaced to the user as a hard failure — the UI just silently gets a stub card.

### `POST /api/knowledge/ingest`

**Input:** `{ personaId: string; text: string; source?: string }` (source defaults `"paste"`).
**Output:** `{ chunkCount: number }` or `{ chunkCount, stub: true }` if `PINECONE_API_KEY` absent.
**External calls:** `chunkText()` (local, ~500-char sentence-aware split) → Pinecone `upsertKnowledgeChunks()` into `persona-knowledge` index, namespace = personaId, integrated inference (`llama-text-embed-v2`), batched at 90 records/call.

Trivial — `users.displayName` / `users.profileBio` read/write for the session user. No external calls.

## 4. Key Components

### `components/Avatar3D.tsx`

Client component, dynamically imported with `{ ssr: false }` from `call/[id]/page.tsx` (TalkingHead touches WebGL/canvas — must never run server-side).

- Loads TalkingHead via `loadTalkingHead()`, which injects `<script type="module" src="/load-talkinghead.js">` (a static file, not a webpack-bundled import) and polls `window.__TalkingHead` (100ms interval, 10s timeout). An importmap for `"three"` (bare specifier, needed by TalkingHead's internal import) is injected first if not already present.
- `avatarUrl` prop is currently **ignored** — `url` is hardcoded to TalkingHead's own demo avatar (`https://cdn.jsdelivr.net/gh/met4citizen/TalkingHead@1.7/avatars/brunette.glb`) as a diagnostic step (see §9).
- Mount effect has `[]` deps — runs once, captures whatever `avatarUrl`/env fallback was current at that instant (moot right now since it's hardcoded, but matters if the hardcode is reverted).
- Mood updates via a separate `[emotion]`-keyed effect calling `head.setMood?.()`.
- No TS types exist for TalkingHead — typed as `any` (`TalkingHeadInstance`) deliberately, not a shortcut.

### `lib/audio.ts` → `AudioQueue`

**Not a class.** An `interface AudioQueue { add, onended, onBuffer, stop, clear }` plus a `createAudioQueue(ctx: AudioContext): AudioQueue` factory closure. This shape is load-bearing — both `chat/[id]/page.tsx` and `call/[id]/page.tsx` depend on it, and it's been proposed-and-rejected as a class rewrite twice (see §11).

```ts
export interface AudioQueue {
  add: (buffer: AudioBuffer) => void;
  onended: (cb: (() => void) | null) => void;
  onBuffer: (cb: ((buffer: AudioBuffer) => void) | null) => void; // Phase 6: drives avatar lip-sync
  stop: () => void;   // hard stop, drops queue
  clear: () => void;  // soft stop, finishes current then stops
}
```
`onBuffer` fires synchronously inside `add()`, before the buffer is pushed — `call/[id]/page.tsx` registers it once in `getAudioQueue()` to forward every buffer to `headRef.current?.speakAudio?.()`.

### `app/call/[id]/page.tsx` — state machine

```
ConvState = "idle" | "listening" | "thinking" | "speaking"
```
See §9 (state machine diagram) in the continuation doc for the full picture — key mechanics:
- Mic (`getUserMedia`/`MediaRecorder`) is lazily created on first click (`ensureMicReady()`), then **never torn down** for the rest of the call — only `sendAudioRef.current` (a boolean gate) controls whether captured chunks get forwarded to the Deepgram WebSocket. This is required for barge-in: the recorder must already be live when state flips to `"speaking"`.
- Click-to-toggle: click 1 (idle→listening) starts forwarding; click 2 (listening→thinking) stops forwarding, sends Deepgram a `{"type":"CloseStream"}` control message, and starts an 800ms fallback timer that submits the last interim transcript (`accTranscriptRef`) if no real Deepgram final arrives in time.
- `handleFinalTranscript` accepts finals while state is `"listening"` **or** `"thinking"` (not just `"listening"`) — necessary because the second click sets `"thinking"` synchronously, before Deepgram's actual final message arrives a beat later.
- Barge-in: a final arriving while `"speaking"` aborts the in-flight turn (`AbortController`), stops the audio queue, calls `headRef.current?.stopSpeaking?.()` and `?.streamInterrupt?.()` (both unconfirmed method names, hedged), and submits the interrupting transcript as a new turn.
- `didInitRef` guards the mount effect's body against double-execution (paired with `reactStrictMode: false` in `next.config.mjs` — the guard alone isn't sufficient under Strict Mode, see §11).

## 7. RunPod Worker Flow (`runpod-worker/handler.py`)

```
Input:  { text, voice_b64, exaggeration=0.5, cfg_weight=0.5, temperature=0.8 }
Output: { audio_base64, sample_rate } | { error: string } | { status:"ok", note:"..." } (startup test)
```

1. Model loaded once at **module level** (`ChatterboxTTS.from_pretrained(device=...)`) — not inside `handler()` — so it's loaded at worker boot, not per-request.
2. `handler(job)`:
   - Empty/near-empty `voice_b64` (< 50 chars after stripping `=`) → returns `{status:"ok", note:"startup test passed"}` immediately (keeps the worker alive for RunPod's health check without needing a real reference clip).
   - Strips `data:...;base64,` prefix if present (defense-in-depth; `/api/tts` also strips it, but the worker doesn't trust the caller).
   - Fixes base64 padding as **two separate statements** — `padding_needed = (4 - len(x) % 4) % 4` then `x += "=" * padding_needed` — combining them on one line risks Python parsing `%` as string-formatting instead of modulo.
   - Detects source format by magic bytes (`RIFF`→wav, `OggS`→ogg, ID3/frame-sync→mp3, else assumes webm).
   - `ffmpeg` converts to 16kHz mono s16 WAV (Chatterbox's expected reference format).
   - `model.generate(text, audio_prompt_path=wav_path, exaggeration, cfg_weight, temperature)` — **no** `generate_stream()`, it doesn't exist on this model.
   - Encodes output tensor to base64 WAV via `soundfile`.
   - `finally` block always cleans up temp files.
3. Bottom of file (**exact structure matters**, this was a real production bug):
```python
if os.environ.get("RUNPOD_ENDPOINT_ID"):
    pass  # running on RunPod — skip local test, fall through to start
else:
    # local test using test_input.json-shaped payload, prints result
    ...
runpod.serverless.start({"handler": handler})  # UNCONDITIONAL, module level, last line
```
`runpod.serverless.start()` must never be nested inside an `if __name__ == "__main__"` guard — RunPod's container entrypoint and local `python handler.py` both hit this file the same way, and gating the start call behind the wrong condition previously caused the worker to exit 0 right after the startup test with `runpod.serverless.start()` never executing.

**Pinned dependency:** `transformers==4.46.3` — do not upgrade, breaks Chatterbox's Llama backbone silently. `requirements.txt` has a duplicate-entry warning built into the RunPod LLM setup docs (not this worker) — not applicable here, no vLLM in this repo.

## 9. Active Bugs / Known Gaps (exact locations)
